Project Overview

The project is a single-file Pygame simulation in main.py. The active code spawns colored squares with random size and velocity, updates motion each frame, applies flee behavior away from larger nearby squares as well as chase behavior for larger squares, and renders FPS text.

Primary Loop

run -> handle_events -> update -> draw -> tick

Core Objects

Game class owns window, clock, font, and square list. Square class owns position, speed, size, and color.

Inputs and Outputs

Keyboard ESC or window close as input. Visual animation and FPS text as output.

Key Files

main.py: current runnable game with flee and chase behaviors implemented.

Testing Notes

No automated tests are present. Current verification is manual runtime behavior. Suggested tests:

  • Boundary bounce correctness for each wall and corner cases.
  • Speed clamp behavior by size after jitter and flee updates.
  • Stability test for 5 to 10 minutes to detect drift or frame drops.
  • Unit test for target_speed monotonic relation with size.

Gaps and Assumptions

  • Assumes pygame and display context are available on the runtime machine.
  • No profiling logs in repo, so performance notes are static code analysis only.
  • main(old).py appears non-runnable as-is due to update signature mismatch.

Patterns and Code Smells

Good Patterns

✅ Separation of Concerns

The Game class orchestrates loop stages while Square encapsulates motion and rendering.

Basics: Each class has a clear job.

Engineering Insight: Lower cognitive load when debugging event loop vs movement logic.

Architecture Insight: This structure can scale to more entity types later.

✅ Named Constants

Behavior values like FPS, flee radius, and jitter are centralized at top-level constants.

Basics: Easier tweaking and readability.

Engineering Insight: Reduces hidden magic numbers spread through methods.

Architecture Insight: Enables future config loading from file or CLI.

✅ Type Hint Coverage

List[Square] and pygame.Surface annotations communicate intended data contracts.

Basics: Readers know expected parameter and return types.

Engineering Insight: Better IDE support catches mismatches early.

Architecture Insight: Helps maintain stable interfaces between objects.

Potential Issues

⚠️ O n2 Neighbor Scan

Every square loops over every other square each frame to compute flee vectors.

Basics: Work grows quickly as square count increases.

Engineering Insight: Frame drops become likely with larger populations.

Architecture Insight: Spatial partitioning can isolate local neighbors.

⚠️ Legacy File Drift

main(old).py includes stale constants and an update call mismatch, increasing confusion risk.

Basics: Students may run wrong file and hit errors.

Engineering Insight: Contradictory examples slow debugging and onboarding.

Architecture Insight: Archive old code under clear version folders or tags.

Performance Analysis

🚀 Wins

  • Clock tick caps the frame rate, preventing runaway CPU in idle scenes.
  • Rendering is simple rectangles, which is relatively cheap for pygame.

🐢 Risks

  • Neighbor flee computation is O n2 each frame and dominates as entities grow.
  • Each square computes square root for nearby objects, adding arithmetic cost per interaction.
Performance Score: 3 / 6 | Verdict: acceptable for 20 squares, risky if scaled significantly.
